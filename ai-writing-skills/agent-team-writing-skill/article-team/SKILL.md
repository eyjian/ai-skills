# AI 技术文章编写 Agent Team

这是一个真正的 Agent Team——5 个 Agent 以网状拓扑协作，可以互相直接对话、自主决策。

## 使用方式

输入 `/article-team` 加上选题方向，或现有文章文件路径 / 改稿需求，即可启动团队。
如果给出的是明确 `.md` 文件路径，团队会优先按“现有文章”处理，再根据动作词判断是先重审、先处理结构，还是直接润色收尾。

示例：
```
/article-team Agent 编排模式对比
/article-team 最近 AI 编程有啥值得写的
/article-team 重审 docs/agent-orchestration.md
/article-team 请润色 article.md，重点降低 AI 味
/article-team 优化 docs/agent-orchestration.md 的开头，让它更像真人作者写的
/article-team 重构 docs/agent-orchestration.md 的章节顺序，先给结构级改稿方案
/article-team 处理 docs/agent-orchestration.md：先审结构，再改稿，最后做终稿润色
```

## 支持三类入口场景

| 场景 | 典型输入 | 默认起点 | 协作路径 |
|------|---------|---------|---------|
| 新稿创作模式 | 主题方向、选题想法 | `scout` | `scout → architect → writer → reviewer → polisher` |
| 旧稿重审 / 回炉模式 | `.md` 文件路径、“重审这篇文章”、“回炉改一版”、“检查这篇” | `reviewer` | `reviewer → (architect / writer 可插入) → polisher` |
| 旧稿直接润色模式 | `.md` 文件路径 + “只润色 / 去 AI 味 / 发布前打磨 / 优化开篇 / 改首段 / 终稿处理” | `polisher` | `polisher → (必要时 reviewer / architect / writer) → 完成` |

说明：
- **新稿创作模式**：适合从 0 到 1 生成文章
- **旧稿重审 / 回炉模式**：适合对已有 Markdown 文章做重审、改稿，以及结构级大改
- **旧稿直接润色模式**：适合文件已基本成型，只需做统一术语、降 AI 味、优化开篇、发布前收尾
- 如果用户明确说“重构结构 / 调整章节顺序 / 先给结构方案 / 大改章节”，团队会把它当作旧稿回炉里的**结构优先任务**处理
- 如果用户明确说“只润色”，团队可直接从 `polisher` 进入；但一旦发现技术问题，`polisher` 需回退给 `reviewer`

## 旧稿输入优先级速查

| 用户输入特征 | 默认判定 | 默认起点 | 说明 |
|-------------|---------|---------|------|
| “重审 / 复审 / 检查这篇 / 回炉改一版” | 旧稿重审 / 回炉模式 | `reviewer` | 先审再决定是否联动 `architect` / `writer` / `polisher` |
| “只润色 / 发布前打磨 / 降 AI 味 / 优化开篇 / 改首段 / 终稿处理”且文件明确 | 旧稿直接润色模式 | `polisher` | 直接收尾，发现技术问题再回退 `reviewer` |
| “重构结构 / 调整章节顺序 / 先给结构方案 / 大改章节” | 旧稿重审 / 回炉模式（结构优先） | `reviewer` | 团队优先处理结构问题，并尽快联动 `architect` |

## 团队成员

| 成员 | 代号 | 职责 |
|------|------|------|
| 选题侦察员 | scout | 搜索热点，提供选题建议 |
| 大纲架构师 | architect | 设计文章结构和大纲 |
| 初稿写手 | writer | 按大纲撰写 Markdown 初稿 |
| 技术审稿人 | reviewer | 审查准确性和完整性，**自主决定退回** |
| 终稿润色师 | polisher | 最终打磨和格式规范化 |

## 角色视觉系统

为了让多角色对话一眼可辨，统一使用“角色图标 + 彩色方块”作为视觉标识。这样即使在不支持真实文字着色的终端或聊天界面里，也能稳定表达颜色语义。

| 角色 | 固定视觉签名 | 颜色语义 | 含义 |
|------|--------------|----------|------|
| 协调者 | `🧭🟦【main｜协调者】` | 蓝色 | 协调、导航、转述 |
| 选题侦察员 | `🔎🟩【scout｜选题侦察员】` | 绿色 | 搜索、探索、发现 |
| 大纲架构师 | `📐🟪【architect｜大纲架构师】` | 紫色 | 结构、框架、设计 |
| 初稿写手 | `📝🟧【writer｜初稿写手】` | 橙色 | 创作、推进、成文 |
| 技术审稿人 | `🛡️🟥【reviewer｜技术审稿人】` | 红色 | 审查、质疑、把关 |
| 终稿润色师 | `✨🟨【polisher｜终稿润色师】` | 黄色 | 打磨、提亮、收尾 |

## 与 Sub-Agent 流水线的区别

| 维度 | 本 Agent Team | 典型 Sub-Agent |
|------|-------------|----------------|
| Agent 间通信 | 任意成员可直接对话 | 只能发消息给 main |
| 决策权 | reviewer 自主决定退回 | 协调者决定是否退回 |
| 流程 | 动态——可并行、可跳步 | 固定——严格串行 |
| 协调者角色 | 传话 + 启动新成员 | 全局决策者 |

## 可视化特性

为了让使用者直观感受到 Agent Team 的网状协作（而非 Sub-Agent 流水线），运行过程中会展示：

1. **🚀 团队拓扑图**：启动时展示网状连接关系，一眼看出任意成员可直接对话
2. **📨 通信日志**：实时展示 Agent 间的消息流向，包括未经协调者中转的直接通信
3. **📊 状态面板**：在关键节点展示各成员的工作状态和当前动态
4. **📈 协作总结**：完成时展示团队协作统计（总通信次数、直接通信次数、退回次数等）
5. **🗣️ 角色署名发言**：所有展示给用户的正文消息都显式标注来源角色（`main` / `scout` / `architect` / `writer` / `reviewer` / `polisher`），避免对话里只出现模糊的“我”
6. **🎨 角色视觉编码**：每个角色都有固定的图标 + 颜色代理（彩色方块 emoji），即使用户不细读文字，也能迅速分辨当前是谁在发言

## 需要用户确认的节点

### 新稿创作模式

1. **选题确认**：scout 给出方案后
2. **大纲确认**：architect 设计完后
3. **终稿确认**：polisher 润色完后

### 旧稿任务模式

1. **终稿确认**：polisher 完成收尾后
2. **结构级改稿确认（按需）**：如果用户输入已明确强调“重构结构 / 大改章节”，或 reviewer / architect 判断需要较大结构调整，再由 main 向用户确认改稿方向
